# Agent Prompt Construction

Principles governing how c-thru agent prompts are written, sized, and tested. Prompts are the contract between the driver (deterministic) and the agent (LLM) — they must be parseable, tier-appropriate, and failure-aware. The 10 foundational principles emerged from comparing c-thru's trigger-based design against BMAD/SDD and analyzing what makes agent prompts reliable across tiers.

- **From Session 1558f542:** 10 foundational principles: (1) Role specificity matches model tier — 7B gets ~3 sentences; cloud judge tolerates framing + tradeoffs. Mismatch = wasted tokens or under-reasoning. (2) Output schema IS the contract — parseable STATUS blocks make the driver deterministic. (3) Forcing functions before verdict — require specific answers ("What does criterion 1 require? Does output satisfy? yes/no + evidence") before the model emits output; converts optimism to evidence. (4) Hard vs. soft language matches hard vs. soft constraints — MUST/NEVER for invariants, Prefer/Favor for heuristics; don't mix. (5) Frame first, constrain second, example last — position-weighted attention means schema-first prompts produce shallow outputs. (6) Eliminate scope ambiguity — every prompt says what's OUT of scope; learned from spec-kit issue #896 where `/speckit.constitution` both amended constitution AND generated code. (7) Path-based inputs — extends c-thru's existing invariant; prompts stay short, model reads the file, context is controlled. (8) Vocabulary consistency — every agent uses `success_criteria`, `target_resources`, `depends_on`, `STATUS`, `VERDICT`; renames cause routing bugs. (9) Model the failure, not just success — name 1-3 specific ways the agent will go wrong + anti-instruction for each; highest-value prompt block. (10) Calibration via logs — every prompt is a hypothesis; patterns emerge only in aggregate (100+ uses).
- **From Session 1558f542:** Token budget ceilings per tier: pattern-coder (1.7-9B local) ~200 tokens, code-analyst (9-27B local) ~500 tokens, deep-coder (27B+ local) ~800 tokens, orchestrator (cloud) ~1200 tokens, judge (cloud) ~1500 tokens, judge-strict (cloud) ~2000 tokens. If a prompt exceeds its tier ceiling, refactor rather than expand — excess tokens indicate missing structure in the prompt.
- **From Session 1558f542:** Testing discipline for prompts: (1) red-team inputs per named failure mode before rollout, (2) A/B testing via `/compare-prompts` skill, (3) calibration audit after 100 real uses, (4) prompts versioned in git with PR review for changes. This treats prompts as testable hypotheses, not static configurations.
- **From Session 1558f542:** Load-bearing prompt patterns documented: decider prompt (forcing functions + three named failure modes — lazy-restart, lazy-uplift, lazy-accept + exclusive VERDICT schema), worker CONFIDENCE self-check (concrete rubric + pre-return one-failure-fix forcing function for medium/low confidence), cloud-uplift prompt ("draft as hypothesis" framing — neither deference nor adversarial rewrite), cloud-restart prompt (deliberately short, no mention of prior attempts — enforced by grep assertions in code + unit tests that "draft", "prior", "previous", "attempt" never appear in clean-mode prompts).
- **From Session 1558f542:** Confidence self-report validation principle: before adding self-assessment fields to a prompt, verify externally that the signal is discriminable — run a post-hoc judge over existing outputs; if the judge can't separate pass/fail from text, the in-prompt self-report won't either. Self-reports without external validation are ungrounded hypotheses, not measurements.

- **From Session 69bfbcd1:** Structural testing gap: 4 agents (integrator, doc-writer, discovery-advisor, planner-local) are missing RECUSE blocks. The existing `c-thru-contract-check.sh` validates dangling references and missing prompt keys but does NOT validate prompt content (RECUSE, STATUS field conformance, escalation chain references). New test `test/agent-prompt-structure.test.sh` planned to fill this gap.

- **From Session a82cecbf:** Self-directed directives principle: when adding a behavioral loop to an agent (e.g. lint verification), a brief directive in the agent prompt ("run available linters, fix errors, skip if unavailable") is preferred over a procedural section with elaborate guard logic. The agent prompt IS the system prompt for each dispatch — brief directives let the model exercise judgment rather than follow brittle scripting. "A missing linter is not failure — skip it" is more robust than `command -v` guard prose.

- **From Session b1731578:** Structural testing gap resolved: static linter (`test/agent-contract-static.test.js`, 94 assertions) and live proxy test (`test/agent-contract-live.test.js`) now validate agent prompt conformance. Fail-closed roster cross-references `agent_to_capability` config — new agents without roster entries fail immediately. Spec-gap agents (integrator, doc-writer, discovery-advisor, planner-local) flagged as warnings, not failures.

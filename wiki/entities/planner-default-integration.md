# Planner Default Integration

`/c-thru-plan` is surfaced as the default planning path via a soft-redirect architecture: an `EnterPlanMode` PreToolUse advisory hook emits a hint to stderr whenever native plan mode is entered, suggesting `/c-thru-plan <intent>` for multi-wave feature work. The hook never blocks — native plan mode continues to function identically, which is required for `review-plan`'s Phase 8 `ExitPlanMode` flow. Components: hook script (`tools/c-thru-enter-plan-hook.sh`), command shortcut (`~/.claude/commands/cplan.md`), skill subcommand (`c-thru-config planning`), and install wiring (`install_cplan_command()` + `install_planner_hint_hook()`).

- **From Session 2a5c31f5:** Non-blocking advisory was the only viable design — hard-blocking `EnterPlanMode` would break `review-plan`'s Phase 8 contract (it enters plan mode so ExitPlanMode can be gated by the review marker). A bypass-credential approach adds coupling; advisory-only is strictly simpler. The hook uses `set +e` / always-exit-0 posture matching the hook-safety-posture pattern for control-flow hooks.
- **From Session 2a5c31f5:** Global-scope constraint: `settings.json` is user-global (`~/.claude/settings.json`), so the `EnterPlanMode` hook fires in every Claude Code session on the machine, not just c-thru projects. `install.sh` prints a one-time disclosure on first registration. Idempotent re-runs suppress the notice. Three opt-out paths: per-session env `CLAUDE_ROUTER_PLANNER_HINT=0`, persistent toggle `/c-thru-config planning off`, and install-time skip via `planner_hint: false` in overrides.
- **From Session 2a5c31f5:** `c-thru-config planning` uses natural-language intent inference instead of strict `on|off|status` keywords — "turn off"/"disable"/"silence" → disable; "enable"/"activate"/"add" → enable; "status"/"what"/"show" → status; ambiguous → asks a clarifying question before acting. This was a user-requested correction from the initial strict-keyword design.
- **From Session 2a5c31f5:** CLAUDE_PROFILE_DIR precedence bug found by code review: `install.sh` used `CLAUDE_DIR` for path resolution while the hook script and SKILL.md used `CLAUDE_PROFILE_DIR`. For users overriding `CLAUDE_PROFILE_DIR`, the path written into `settings.json` and the path the `planning` subcommand greps for would diverge — hook removal, status, and idempotency all break silently. Fix evolved in two passes: first a three-way `CLAUDE_PROFILE_DIR → CLAUDE_DIR → $HOME/.claude`, then simplified to `${CLAUDE_PROFILE_DIR:-$HOME/.claude}` — the middle `CLAUDE_DIR` tier was dead code that could never differ from the default.
- **From Session 2a5c31f5:** `cplan.md` intent routing: the initial implementation used `args="$ARGUMENTS"` passed to `Skill()`, but c-thru-plan has no receiver for args — it derives intent from conversation context (Phase 0: "Compute SLUG from the user intent"). Fix: intent is placed as inline text before the `Skill()` call so c-thru-plan reads it from conversation context, which is its actual mechanism.
- **From Session 2a5c31f5:** Review-fix findings (pass 3): (1) `planning toggle` now has a concrete bash block reading `count` and `hint_pref` in one pass, then disables if on or enables if off — no LLM improvisation needed. (2) `planning status` now reflects `CLAUDE_ROUTER_PLANNER_HINT=0` env, reporting `effective: suppressed` instead of the misleading `effective: on`. (3) `install_planner_hint_hook` opt-out check moved first — `planner_hint: false` in overrides takes precedence regardless of whether the hook is already registered, closing the edge case where a manually re-registered hook with an active opt-out would skip the opt-out path.
- **From Session ddd426f8:** Review-plan found 5 substantive issues before convergence: (1) Q-C38 — need a spike to verify `EnterPlanMode` PreToolUse hook actually fires before building the hook script (Section 0 added to plan); (2) Q-C33 — `planner_hint` is NOT handled by `model-map-edit.js` (`applyUpdates` silently drops unknown keys), so `c-thru-config planning` must use direct `jq` write to overrides; (3) Q-C28 — hook fires in all sessions machine-wide, `install.sh` must disclose this on first registration; (4) Q-G22 — cross-phase dependency markers needed (Section 1 Outputs → Section 3/4 Pre-checks); (5) Q-G13 — per-section verify+commit cadence required.

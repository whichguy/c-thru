# Cascade Scope Contraction

A self-review (§19 of the BMAD/SDD comparison plan) identified 12 real weaknesses in the original 8-expansion cascade proposal and deliberately contracted the committed scope to a single change: adding a CONFIDENCE field to worker STATUS blocks plus a logging hook for calibration. The principle: every downstream proposal (decider, cascade, converger, best-of-N) depends on the CONFIDENCE signal being trustworthy — if it isn't, the whole cascade is fiction. Test the load-bearing assumption before building on it.

- **From Session 1558f542:** 12 identified weaknesses in the original plan: W1 scope explosion (8 expansions is a prospectus, not a plan), W2 CONFIDENCE load-bearing but uncalibrated (cold-start means cascade decisions are noise until proven), W3 decider introduces net-new cloud hop (math unclear on whether decider cost is offset by saved hops), W4 verdict-distribution targets fabricated (no empirical basis, should measure and use observed distribution), W5 cascade budget arbitrary (should start unlimited, set to P95+1), W6 premature generalization (§11 extends cascade to 4 roles before implementer cascade is proven — YAGNI), W7 anchor-prevention grep assertion is fragile (clean-mode digests contain "prior-wave context" that false-positives; use whitelist construction not denylist), W8 reviewer-fix interaction underspecified (decider at round 2 + possible cloud = up to 6 hops, regression for simple items), W9 CLAUDE.md re-read may already happen via recon.md synthesis, W10 plan file too long to act on (~1000 lines), W11 decider prompt template ~800 tokens + inline context = 3-5k real tokens, W12 security-reviewer-local conflict resolution unspecified.
- **From Session 1558f542:** Revised Wave-1 scope: single change — land CONFIDENCE field on worker STATUS and instrument logging for calibration. Affected files: `agents/implementer.md`, `agents/reviewer-fix.md`, `agents/test-writer.md`, `agents/scaffolder.md`, `agents/plan-orchestrator.md`, `docs/agent-architecture.md`, `CLAUDE.md`. No decider. No cascade. No generalization. Implementation: edit canonical worker → smoke test parser → roll to remaining workers → update orchestrator logging → update docs → run 3 real plans (≥50 items) → compute calibration → decision gate.
- **From Session 1558f542:** Decision gate after Wave 1: if calibration is usable (flag if >30% of high-confidence items fail verify; adjust threshold from observed distribution), proceed to Wave 2 (build decider). If not, iterate on worker rubrics for one more wave. If still not after 2 gate attempts: abandon cascade design, rely on existing reviewer-fix loop. This explicit kill-switch prevents sunk-cost escalation into a cascade built on an unreliable signal.
- **From Session 1558f542:** Pre-condition check before Wave 1: verify whether `recon.md` synthesis in Phase 1 already ingests CLAUDE.md (if yes, the §5.2 planner.md re-read edit is redundant). Read `skills/c-thru-plan/SKILL.md` Phase 1 Stage 3 before editing `agents/planner.md`.
- **From Session 1558f542:** Pre-Wave-1 experiments replaced the earlier post-hoc judge advisory with concrete zero-code-change gates. Experiment A (local rubric calibration, ~15 min): prompt a local model (devstral-small or qwen3.5:9b) with the §12.1 rubric against two known-good/known-bad worker outputs; gate is categorical match at n=2. Experiment B (cloud materiality check, ~10 min): same two outputs with cloud judge tier; 2×2 decision table (local/cloud × good/bad) determines whether cloud decider is optimization or requirement. Both run before any branch is opened — read-only probes that gate Wave-1 entry.
- **From Session 1558f542:** Categorical agreement for small n: percentage thresholds (>80%/<60% agreement) are meaningless at n=2 judges — replaced with categorical language (agree/contradict/mixed). Decision rule: 2-alike = go, 0-alike = no-go, 1-alike = iterate. Caught by review-plan Gate 1 on convergence pass 2; applies to any small-n judge agreement protocol.
- **From Session 1558f542:** 7 key architectural decisions settled by this session: D1 (test load-bearing assumption first — Experiments A+B gate everything), D2 (no prior wave data — Wave-1 IS the data-generation step, AUC check runs mid-Wave-1 after plan 1), D3 (parser safety gates everything — S0 before branching), D4 (calibration formula excludes null verify.tests_pass items to avoid inflation), D5 (two-run gate persistence at n≈10-20 per worker — single-run flag is hypothesis-generating, not gate-failing), D6 (planner-signals grouping validated — one planner + three signals, ~10-15× lighter than BMAD), D7 (no constitution file — poor evidence, CLAUDE.md sufficient).
- **From Session 1558f542:** Plan consolidated from 2476 lines (full research + 8 expansion proposals) to 331 lines (Wave-1 scope only + appendices) after self-review identified W1 (scope explosion). Review-plan ran 4 passes total, converging on the minimal committed scope. The ~2000 lines of research appendix and cascade design were preserved as Appendices A-C but explicitly deferred pending Wave-1 gate results.

- **From Session a82cecbf:** Wave-2 shipped (PR #34) — the Wave-1 kill-switch gate was implicitly passed (CONFIDENCE field was live and usable from Wave-1 shipping in PR #33). The cascade scope contraction was vindicated: Wave-1's minimal scope (CONFIDENCE + logging) proved the signal was load-bearing enough to build on, and Wave-2's implementation (self-recusal chain, uplift-decider, cloud workers) layered cleanly on top.

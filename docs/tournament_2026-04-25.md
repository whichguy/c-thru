## Key Learnings & Surprises

Observations from this tournament that were unexpected, counter-intuitive, or meaningfully changed our understanding of these models.

### Scoring System: What the New Criteria Revealed

- **Old regime (regex/JSON-structure only): 58% of responses scored q=5.** New regime: mean q=4.25/5.0 with genuine spread — the redesigned prompts and execution-based + criterion scoring actually discriminate.

- **C1 (LRUCache implementation, execution-tested): universal q=3 ceiling.** Every single model — from 6GB vision models to 120GB large generals — passed only a subset of the 5 hidden test cases. No model achieved q=5. This proves the execution scorer is working: correct O(1) LRU with proper eviction order is genuinely hard to produce in a single zero-shot pass.

- **C2 (binary search bug fix, execution-tested): hard split between models.** 18 models passed (q=5), 9 failed (q=1). Pass: DSV4-Flash-Cloud, DeepSeek-R1-14B, DeepSeek-R1-32B, Devstral-2, Devstral-Small-2, GLM-4.6.... Fail: GPT-OSS-20B, Kimi-K2.6, Phi-4-Reasoning, Phi-4-Reasoning-Plus, Qwen3-VL-8B.... The split is NOT correlated with model size — see surprises below.

- **D2 (Go errgroup leak, criterion-tested): most models pass, but surprisingly qwen3-coder:30b (primary coder, 123 t/s) scored q=2 while qwen3-vl:8b (8B vision model) scored q=5.** The errgroup criterion tests whether a model identifies that the goroutine blocks on send after context cancellation — the vision model's training apparently covered this Go pattern better than the dedicated coder fine-tune.

### Surprises & Counter-Intuitive Results

- **SIZE INVERSION: qwen3.6:27B outperforms qwen3.6:35B across coder, debugger, and generalist roles.** On cross_role_coder: 27B avg=4.5 vs 35B avg=3.5. The 27B also passes C2 binary search (q=5) while the 35B fails (q=1). More tokens ≠ better code reasoning at this scale. The 27B runs at 19 t/s vs 60 t/s for the 35B — so not a speed winner — but for quality-critical tasks the smaller model wins.

- **EXTENDED THINKING HURTS: phi4-reasoning:plus scored worse than phi4-reasoning:latest on judge calibration prompts.** phi4:latest J1=q=5, J2=q=5, J4=q=5. phi4:plus J1=q=1, J2=q=1, J4=q=5. phi4:plus took 464–1137 seconds per prompt (vs 20–38s for :latest) and actively second-guessed correct scoring verdicts after extended reasoning. Conclusion: extended thinking chains are counterproductive for structured criterion-evaluation tasks.

- **MoE DOMINANCE: Gemma-4-26B-MoE runs at 103 t/s vs Gemma-4-31B at 24 t/s — 4.3× faster — at identical quality (both q=5.0 on fast_generalist).** The 26B MoE uses 17GB RAM vs 19GB for the dense 31B. The MoE architecture is a strict win here: less RAM, far faster, same quality.

- **AGENTIC SPECIALIST DISAPPOINTS: devstral-2 (74GB, primary agentic_coder) averaged q=3.2 on its primary role vs devstral-small-2 (15GB) at q=3.1.** The 74GB model consumes 5× the RAM for a 0.1 point quality difference. On A1 (CI investigation plan), the large model scored q=1 while the small model scored q=2 — bigger ≠ better agentic planning. Both models struggled with criterion-scored agentic tasks (A1/A2/A5 all below q=3), suggesting the 'agentic' fine-tuning doesn't help with diagnosis-first behavior.

- **CODING FINE-TUNE QUANTIZATION EFFECT: qwen3.6:35b-coding-nvfp4 and -mxfp8 pass C2 (q=5) while the bf16 (full-precision) variant fails (q=1).** The base 35B also fails. The NVfp4 quantization runs at 124 t/s (vs 70 t/s for bf16) AND produces better code. This suggests the NVfp4/MXfp8 quantization pathways interact with the coding fine-tune in a way that changes which solution tokens are sampled — not purely a quality loss.

- **BEST JUDGE IS THE LOGIC SPECIALIST: phi4-reasoning scored q=5 on all four judge prompts (J1–J4), the only model to achieve this.** Every other model dropped on J4 (trajectory evaluation) to q=2–4. phi4's formal reasoning training apparently transfers directly to evidence-chain evaluation. Recommendation: use phi4 as your LLM judge.

- **HEALTH CHECK HALLUCINATIONS: several models reported false OpenAI identities.** 
  - GPT-OSS-120B reported itself as made by OpenAI
  - GPT-OSS-20B reported itself as made by OpenAI
  - Gemma-4-31B reported itself as made by OpenAI
  These models pass functional tests but cannot accurately describe themselves — relevant when model identity matters for provenance tracking.

- **LONG-CONTEXT FAILURE: llama4:scout timed out on LC1 (50K-token prompt) at 300s.** The prompt eval phase alone exhausted the timeout. LC2 returned q=1 (no context). The long_context role requires either a longer timeout (600s+) or a shorter document (20K tokens). llama4:scout's 10M token context window is architectural capability — but practical throughput at 37 t/s limits usable context on local hardware.

- **ORCHESTRATOR ROLE COVERED ONLY BY CLOUD: GLM-4.6 and GLM-5.1 are the only orchestrator-primary models.** No local model was configured as primary orchestrator. GLM-4.6 scored q=4.8 — the best orchestrator — but requires cloud API access. This is a gap in local-only deployments.

### Key gaps surfaced by this table

**`judge` (claude-opus-4-6)** — No tournament score because Anthropic cloud models were not run through Ollama. The local disconnect fallback (`qwen3.6:35b`) scored 5.0 on generalist, which supports it as an offline judge, but no judge-specific prompts (J1–J4) were run against it. Run `qwen3.6:35b` on judge prompts before relying on the offline cascade.

**`orchestrator` (qwen3.6:35b at 128gb)** — Scored `--` because the orchestrator benchmark role only ran GLM cloud models. Qwen3.6:35b's 5.0 generalist score is evidence it can reason well, but orchestrator prompts test structured JSON dispatch (O1–O4) which requires different behavior. The 64gb/48gb tiers use unscored `q3.6:27b-coding-nvfp4`. Gap: run orchestrator prompts against all local candidates.

**`deep-coder` at 64gb/48gb (claude-sonnet-4-6 cloud)** — Falls to cloud immediately when disconnected; no local `deep-coder` fallback below 128gb. If cloud is unavailable, implementer agents receive the local cascade, which falls to whatever the tier's cascade chain allows. This is the highest-risk gap for offline deployments.

**`deep-coder-precise` quantization inversion** — bf16 (full precision, 70GB) scores **lower** (q=3.5) than mxfp8 (q=4.5) and nvfp4 (q=4.5). Full precision is not better for this model family. The bf16 variant also runs at 69 t/s vs 123 t/s for nvfp4 — it is both slower and worse. Do not promote bf16 to `deep-coder-precise` on 128gb; use mxfp8 (38GB, 4.5 quality, 96 t/s) instead. The nvfp4 (22GB, 4.5, 123 t/s) is the correct offline fallback at 48gb+.

**`pattern-coder` and `local-planner` at 64gb/48gb** — Both use `q3.6:27b-coding-nvfp4`, which has no tournament score. These are high-frequency agents (scaffolder, discovery-advisor run every wave). Unscored models in high-frequency positions is a silent quality risk.
